### 2.3 Logging (`logging_config.py`)

### 2.4 Queue / Worker (`queue.py`)

**Job functions:**

1. **`run_evaluation_job(ctx, eval_id_str, defer_count=0)` (L117-238)** — Main evaluation job in 4 phases:
   - Predecessor deferral: if an earlier eval for same asset+SLO is pending, re-enqueues with 2s delay (max 60 defers) (L132-141)
   - Phase 1: Mark running + snapshot, COMMIT (L143-149)
   - Phase 2a: Load SLO/SLI definitions in short read session (L151-165)
   - Phase 2b: HTTP query to adapter + evaluate with baseline repo (L173-194)
   - Phase 3a: Write evaluation results + indicator rows, COMMIT (L196-213)
   - Phase 3b: Write SLI values to hypertable in separate txn to avoid chunk lock deadlocks (L215-223)
   - Then enqueues `finalize_run_job` for parent run (L228-238)

2. **`finalize_run_job(ctx, run_id_str)` (L241-271)** — Finalizes parent `EvaluationRun` if all children are done. Warms heatmap column cache after finalization.

3. **`finalize_sweeper_job(ctx)` (L282-325)** — Cron job (configurable interval, must divide 60). Scans for stuck parent runs whose children are all terminal but parent is still pending/running. Deadlock-safe: never updates child rows.

**`WorkerSettings` (L328-347)** — arq discovery class. Exposes the 3 job functions, cron schedule for sweeper, startup/shutdown hooks, max_jobs, and Redis settings.

**Worker lifecycle:**
- `_worker_startup` (L88-97): configures logging, creates Redis cache + httpx client with connection limits
- `_worker_shutdown` (L100-107): closes Redis and httpx client

**Helper functions:**
- `_has_pending_predecessor()` (L42-60): checks for earlier pending/running eval for same asset+SLO
- `_mark_failed()` (L110-114): marks an evaluation as failed in a fresh session
- `get_arq_pool(request)` (L75-80): FastAPI dependency returning arq pool from `app.state`
- `create_arq_pool()` (L83-85): creates arq Redis pool from config

### 2.6 Session Management (`db/session.py`)

**`_get_engine()` (L20-31)** — Lazy singleton `AsyncEngine`. Uses `pool_size` and `max_overflow` from config. `echo=False`.

**`get_session_factory()` (L34-39)** — Lazy singleton `async_sessionmaker`. `expire_on_commit=False` prevents attribute access errors after commit.

**`get_session(request)` (L42-45)** — FastAPI dependency extracting session from `request.state.session` (set by middleware).

**`get_cache(request)` (L48-50)** — FastAPI dependency returning `RedisCache` from `app.state.cache`, or `None`.

### 2.7 Session Middleware (`db/middleware.py`)

**`SessionMiddleware` (L9-54)** — ASGI middleware managing per-request DB session lifecycle.
- Creates a new `AsyncSession` per HTTP request, stores in `scope['state']['session']` (L37-38)
- Wraps ASGI `send` to intercept `http.response.start` (L40-45):
  - 2xx status: `session.commit()` (L43)
  - 4xx/5xx status: `session.rollback()` (L45)
- On exception: `session.rollback()` in except block (L51)
- Always: `session.close()` in finally block (L53)
- Non-HTTP scopes (WebSocket, lifespan) pass through untouched (L33-34)

### 2.8 Redis Cache (`cache/redis_cache.py`)

**`RedisCache` (L9-37)** — Minimal read-through cache.
- Constructor stores raw `redis` client (typed as `Any`) (L12-13)
- `get_or_load(key, loader, ttl_seconds=None)` (L15-34): checks cache, calls async `loader` on miss, stores result with optional TTL. Handles bytes->str decode. Returns `None` if loader returns `None` (does not cache `None`).
- `invalidate(key)` (L36-37): deletes a key.

### 2.9 Domain Exceptions (`modules/common/exceptions.py`)

Three-level hierarchy:
- `DomainError(Exception)` (L6) — base
- `NotFoundError(DomainError)` (L10) — takes `entity` and `name`, formats as `"{entity} {name!r} not found"`
- `ConflictError(DomainError)` (L19) — takes `entity`, `name`, `reason`, formats as `"{entity} {name!r}: {reason}"`
- `DomainValidationError(DomainError)` (L29) — takes `detail` string

### 2.10 Exception Handlers (`modules/common/exception_handlers.py`)

Four handlers registered on the app:
- `not_found_handler` (L16-18): `NotFoundError` -> 404, `{"detail": str(exc)}`
- `conflict_handler` (L21-23): `ConflictError` -> 409, `{"detail": str(exc)}`
- `integrity_error_handler` (L26-33): SQLAlchemy `IntegrityError` -> 409, generic message. Safety net.
- `domain_validation_handler` (L36-47): `DomainValidationError` -> 422, structured as `HTTPValidationError` with `detail` list containing loc/msg/type.

### 2.11 Method Not Allowed Middleware (`modules/common/method_not_allowed.py`)

**`MethodNotAllowedMiddleware` (L32-92)** — Precomputes route lookup at app start:
- `_literal_allowed_methods`: `dict[path, set[method]]` for routes without path params (L37)
- `_parameterized_routes`: `list[(regex, set[method])]` for parameterised routes (L38)
- On request: if path matches a known route but method is not in allowed set, returns 405 with `Allow` header (L82-89). Unrecognised paths fall through to normal 404 handling.
- Solves Starlette's greedy parameterised route matching (e.g. `PATCH /datasources/tag-keys` being captured by `PATCH /datasources/{name}`).

## 3. Data Flow

### 3.1 Startup Sequence

1. Python imports `main.py` -> module-level execution creates `app = FastAPI(...)` (L58)
2. `SessionMiddleware` added with session factory from `get_session_factory()` (L59) -> triggers lazy engine creation from config
3. Exception handlers registered (L62-65)
4. Routers included (L68-76)
5. `MethodNotAllowedMiddleware` added outermost, snapshots final route list (L88)
6. On first request, `lifespan()` runs:
   - `get_settings()` -> loads config.yaml (already done at import), caches Settings singleton
   - `validate_required()` -> checks TK_DB_PASSWORD, TK_REDIS_PASSWORD, TK_SECRET_KEY
   - `configure_logging()` -> structlog + stdlib setup
   - `create_arq_pool()` -> connects to Redis db_index for job queue
   - Creates cache Redis connection -> wraps in `RedisCache` -> stored on `app.state.cache`

### 3.2 HTTP Request Lifecycle

1. Request enters ASGI stack
2. `MethodNotAllowedMiddleware.__call__()`:
   - Looks up path in literal map, then parameterised routes
   - If path matches but method is not in allowed set -> 405 response, short-circuit
   - Otherwise passes through
3. `SessionMiddleware.__call__()`:
   - Creates `AsyncSession` from factory
   - Stores session in `scope['state']['session']`
   - Wraps `send` to intercept `http.response.start`
4. FastAPI router dispatches to endpoint handler
5. Endpoint resolves `get_session(request)` dependency -> retrieves session from `request.state`
6. Business logic executes (repository calls, domain logic)
7. If domain exception raised:
   - Exception handler converts to appropriate HTTP response
   - `_finalise_then_send` sees non-2xx status -> `session.rollback()`
8. If success:
   - `_finalise_then_send` sees 2xx status -> `session.commit()`
9. `session.close()` in finally block

### 3.3 Worker Job Lifecycle

1. arq discovers `WorkerSettings` class
2. `_worker_startup()`: configures logging, creates `RedisCache` + `httpx.AsyncClient` in `ctx`
3. Job dequeued: `run_evaluation_job(ctx, eval_id_str)`
4. Each DB phase uses `session_factory()` as async context manager -> fresh session per phase
5. Sessions are manually committed/closed (no middleware)
6. On completion, enqueues `finalize_run_job` for parent run
7. `finalize_sweeper_job` runs on cron schedule as a safety net

## 4. Patterns & Conventions

### 4.1 Dependency Injection

- FastAPI dependencies via function parameters: `get_session(request)`, `get_cache(request)`, `get_arq_pool(request)`
- No DI container -- relies on FastAPI's `Depends()` in routers and module-level singletons
- Worker jobs receive dependencies via `ctx` dict (arq pattern)

### 4.2 Session Management Strategy

- **API requests**: middleware creates session, auto-commits on 2xx, auto-rollbacks on error. Endpoints do NOT call `commit()` -- this is handled by the middleware.
- **Worker jobs**: manual session management via `async with session_factory() as session:` blocks. Each phase gets a fresh session and explicitly calls `session.commit()`. This isolates failures -- a phase 3 commit failure does not roll back phase 1's snapshot.

### 4.3 Configuration Pattern

- Two-tier: YAML for non-secrets (committed), env vars for secrets (TK_ prefix)
- YAML loaded at module import time into `_yaml` dict (L41)
- Each sub-settings class reads defaults from `_yaml` at class definition time
- `@lru_cache` on `get_settings()` but properties create new instances on each access

### 4.4 Error Handling

- Domain exceptions are structured (`entity`, `name`, `reason`) for consistent messages
- Lowercase, no trailing period (per codebase convention)
- `IntegrityError` catch-all prevents raw DB errors from becoming 500s
- `DomainValidationError` is formatted as `HTTPValidationError` list structure (not plain string) for OpenAPI conformance

### 4.5 Null-byte Protection

Pervasive defense against `\x00` in strings that would crash asyncpg. Three layers:
- `SafeStr` for simple string fields (schema-level pattern + runtime validator)
- `SafeJsonDict` for flat JSONB dicts (runtime-only, shallow)
- `SafeJsonAny` for nested JSONB structures (runtime-only, recursive)
This converts what would be 500 (asyncpg `UntranslatableCharacterError`) into 422.

### 4.6 Naming Conventions

- ORM models use PascalCase class names, snake_case table names
- All IDs are UUID v4
- Timestamps use `DateTime(timezone=True)` with `server_default=func.now()`
- `created_at` + `updated_at` pattern (updated_at uses `onupdate=func.now()`)
- Soft delete via `hidden_at` / `hidden_by` / `hidden_reason` (annotations only)
- `active` boolean flag for logical deletion (SLO definitions, SLI definitions, SLO groups)
- `fmt: off` / `fmt: on` blocks around column definitions for aligned formatting

### 4.7 Relationship Loading Strategies

- `lazy='joined'` — eager load in same query (SLODefinition -> SLIDefinition, IndicatorResultRow -> SLOObjective)
- `lazy='selectin'` — eager load via separate SELECT IN (SLODefinition -> objectives, SLOEvaluation -> indicator_rows)
- `lazy='raise'` — prevent accidental lazy loading (SLOAssignment -> slo_definition/data_source)
- No relationship on SLIValue -> SLOEvaluation (intentional, prevents accidental loading of hypertable rows)

## 5. Known Issues & Workarounds

### 5.1 Sub-settings Reinstantiation

`Settings` properties (L213-251) create new `DatabaseSettings()`, `CacheSettings()`, etc. on every access. Only `QueueSettings` is cached in `_queue_settings` (L135). The `@lru_cache` on `get_settings()` does not help because the properties are on the instance, not the class. Repeated calls to `get_settings().database` create multiple `DatabaseSettings` objects. This is likely harmless (Pydantic Settings are lightweight) but inconsistent.

## 6. Cross-cutting Observations

### 6.1 Session Ownership Split

The API and worker have fundamentally different session strategies:
- **API**: Middleware-managed, implicit commit/rollback. Endpoints never call `commit()`.
- **Worker**: Manual session management with explicit commits per phase. This is necessary because worker jobs span multiple independent transactions.

This split is well-designed -- the middleware pattern keeps API code clean, while worker jobs get the fine-grained control they need for multi-phase operations.

### 6.2 Monolithic Models File

All 21 ORM models live in a single `db/models.py` file (711 lines). This is a conscious choice -- it keeps the declarative base in one place and avoids circular import issues. However, it makes the file dense.

### 6.3 Check Constraints as Domain Enforcement

The codebase uses PostgreSQL check constraints extensively:
- XOR constraints on annotations (exactly one parent) and assignments (asset_id XOR asset_group_id)
- Status enums enforced at DB level (`'pending','running','completed','failed','partial'`)
- Result enums enforced at DB level (`'pass','warning','fail','error' OR NULL`)
- Ingestion mode enforced (`'pull','push','file'`)

This provides defense-in-depth -- even if application code has a bug, invalid data cannot enter the database.

### 6.4 Denormalization Strategy

`SLIValue` deliberately denormalizes `asset_name`, `evaluation_name`, and `os_tag` to avoid joins in Grafana SQL queries. The composite PK includes the partition key (`eval_start`) as required by TimescaleDB.

### 6.5 Schemathesis-Driven Hardening

Multiple design decisions are driven by schemathesis (API contract testing tool):
- `SafeStr`, `SafeJsonDict`, `SafeJsonAny` — prevent 500s from null bytes
- `StrictQueryBool` — prevent lenient bool parsing from causing unexpected 422s
- `IntNotBool`, `FloatNotBool` — prevent booleans from being accepted as numbers
- `TagKey`, `TagValue` — constrained patterns so fuzzers generate realistic payloads
- `_inject_property_names_pattern` — OpenAPI schema patching for hypothesis-jsonschema
- `MethodNotAllowedMiddleware` — prevents parameterised routes from swallowing wrong-method requests

This is an unusually thorough level of API contract correctness.

### 6.6 Cache Layer Simplicity

`RedisCache` is intentionally minimal -- just `get_or_load` and `invalidate`. It does not cache `None` values (cache-miss on missing data), does not support batch operations, and has no serialisation layer (callers handle JSON encoding). The TTL configuration in `CacheTTLSettings` shows per-endpoint granularity with sensible defaults (30s for lists, 300s for detail, 7 days for heatmap columns).

### 6.7 Worker Predecessor Deferral

The `_has_pending_predecessor` check (L42-60 in queue.py) implements ordering guarantees for evaluations of the same asset+SLO. This prevents out-of-order baseline comparisons. The max defer count of 60 with 2-second delay gives a 2-minute window before giving up. Errors in the predecessor check are swallowed (returns `False`) so evaluations proceed even if the check fails.

### 6.8 Finalization Safety Net

The finalize architecture has two paths:
1. **Fast path**: Each child evaluation enqueues `finalize_run_job` after completion
2. **Sweeper**: Cron job scans for stuck parent runs periodically

The sweeper exists because the fast path can fail silently (e.g., Redis unavailable when enqueuing finalize). The comment at L226-233 explains why `_job_id` is not used for deduplication -- arq's result key persistence would prevent re-enqueueing after a premature execution.
