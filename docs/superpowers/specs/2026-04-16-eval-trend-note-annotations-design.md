# Evaluation Trend Chart — Note Annotations

**Status:** Draft
**Date:** 2026-04-16
**Branch:** `feat/asset-meta-timeline` (follow-up feature on same branch or next)

## Problem

The evaluation detail page renders a `MetricTrendBlock` (ECharts) per indicator
showing historical values. Existing notes (`Annotation`) attached to evaluation
runs already appear in `EvaluationNotesSection` and as hover indicators over
the heatmap (`NoteIndicatorRow`), but they do **not** appear on the trend
charts themselves. A user reading a trend has to manually cross-reference
notes. This design adds per-category opt-in chart annotations so informative
notes (e.g. `info`, `failure`, `investigation`) surface directly on the trend,
while operational notes (`re-evaluation`, `override`) stay off the chart.

Override state is surfaced via a dedicated visual treatment on the existing
data point (a gray ring), independent of the category system.

Asset-meta information (releases, deployments) is explicitly **out of scope**;
it will get its own timeline feature.

## Goals

- Notes categorised as chart-visible appear as a pin with a short label at
  their evaluation's x-position on every trend chart on the eval page.
- A new `AnnotationCategory` entity drives visibility and appearance.
- Simple CRUD UI for managing categories.
- Per-chart toggle to hide all note annotations when the user wants a clean
  view.
- Overridden evaluation points get a distinctive ring on the existing data
  point (no pin, no category).

## Non-goals

- Asset-meta timeline (releases/deployments) — separate feature.
- Per-asset-group or per-asset category scoping. Categories are global.
- Category-based filtering in the notes list or heatmap.
- Free-text icons or per-category icon configuration (colored pin only for v1).
- Chart annotations on anything beyond `MetricTrendBlock` (heatmap already has
  `NoteIndicatorRow`; `MultiSeriesChart` is a future adopter via the shared
  helper).

## Data model

### New table: `annotation_category`

| Column | Type | Notes |
|---|---|---|
| `id` | UUID (PK) | |
| `name` | `str`, unique, not null | Stable key (e.g. `info`, `failure`, `investigation`, `re-evaluation`, `override`) |
| `label` | `str`, not null | Display label shown on chart pill. ≤12 chars enforced at API + UI. |
| `color` | `str`, not null | Token from a restricted palette (see §Palette). |
| `show_on_graph` | `bool`, not null, default `true` | Drives chart visibility. |
| `is_system` | `bool`, not null, default `false` | `re-evaluation`, `override` are system rows — cannot be deleted, `name` cannot be edited, `show_on_graph` can still be toggled. |
| `created_at` / `updated_at` | timestamptz | Standard audit columns. |

Sort order in dropdown and management UI: by `name` ascending.

### Seeded rows

| name | label | color | show_on_graph | is_system |
|---|---|---|---|---|
| `failure` | Failure | `red` | `true` | `false` |
| `info` | Info | `sky` | `true` | `false` |
| `investigation` | Investigation | `amber` | `true` | `false` |
| `re-evaluation` | Re-eval | `gray` | `false` | `true` |

Only `re-evaluation` is a system row because it is the only category written
automatically by backend workflows (`re_evaluation_service.py`). The override
visual is driven by `Evaluation.originalOutcome != null` and has no
corresponding annotation or category.

`info` is the default category for new notes in `AddNoteForm`.

### Palette

A small, theme-safe palette. Category `color` accepts only these tokens; the
UI resolves each to a CSS custom property that already adapts to the active
theme:

`sky`, `green`, `amber`, `red`, `purple`, `pink`, `slate`, `gray`

### `annotation.category` → `annotation.category_id`

- New column `annotation.category_id UUID NOT NULL REFERENCES annotation_category(id)`.
- Drop the old `annotation.category TEXT` column.
- Because there are no prod customers, migration history is regenerated via
  `scripts/db-regen-migrations.sh` (one autogenerated `001_initial_schema.py`
  + the manual `002` timescale seed). The seed migration (`002`) is extended
  to insert the five seeded category rows.
- Backfill for any existing local dev data is handled by `dev-start.sh`
  re-running the seed; no in-place migration is required.

### Repositories

`AnnotationRepository` signature changes: `category: str | None` parameter is
replaced by `category_id: UUID` (required). `re_evaluation_service.py` is
updated to look up the system `re-evaluation` category by `name` once at
startup (cached) and pass its id.

New `AnnotationCategoryRepository` with `list_all()`, `get_by_id()`,
`get_by_name()`, `create()`, `update()`, `delete()`. `delete()` blocks if
`is_system=true`; otherwise cascades via a two-step operation: move all
referencing annotations to the `info` category, then delete the row.

## API

New endpoints under `/note-categories`:

| Method | Path | Body | Returns |
|---|---|---|---|
| GET | `/note-categories` | — | `list[AnnotationCategoryRead]` |
| POST | `/note-categories` | `AnnotationCategoryCreate` | `AnnotationCategoryRead` |
| PATCH | `/note-categories/{id}` | `AnnotationCategoryUpdate` | `AnnotationCategoryRead` |
| DELETE | `/note-categories/{id}` | — | `204` with usage count in `X-Reassigned-Annotations` header |

Schemas (Pydantic, `StrictInput` for request bodies):

- `AnnotationCategoryRead`: all columns (excluding audit timestamps).
- `AnnotationCategoryCreate`: `name`, `label` (≤12 chars), `color` (palette
  enum), `show_on_graph`.
- `AnnotationCategoryUpdate`: partial — `name` rejected for system rows.

Annotation schemas:

- `AnnotationRead` gains `category_id: UUID` and `category: AnnotationCategoryRead`
  (embedded) and drops the text `category` field.
- `AnnotationCreate` gains `category_id: UUID` (required); old `category` text
  field removed.

### `MetricTrendBlock` annotations

New shared helper `lib/chartAnnotations.ts`:

```ts
export function buildNoteMarkPoint(args: {
  trendPoints: TrendPoint[]
  annotationsByEvalId: Map<string, Annotation[]>
  categoriesById: Map<string, NoteCategory>
  chartWidth: number
}): MarkPointOption
```

Returns an ECharts `markPoint` object. Behaviour:

1. For each trend point with one or more annotations whose category has
   `show_on_graph=true`:
   - Compute dominant category = the chart-visible annotation with the
     alphabetically-first category `name` (deterministic, no config needed).
   - Emit one markPoint item at `(xIndex, pointValue)`:
     - `symbol`: custom SVG teardrop path, size 18×14 px, offset `[0, -12]`
       above the point, fill = category color (resolved to CSS var).
     - `label.show`: true if `chartWidth / visibleEvalCount >= 40`
       (dense-mode fallback — icons only when too dense).
     - `label.formatter`: `category.label` + ` (n)` if `visibleCount > 1`.
     - `tooltip.formatter`: renders the full `TooltipNoteList` component's
       output (extracted to a shared renderer usable from ECharts' HTML
       tooltip formatter).

2. markPoint is included in the ECharts `series[0].markPoint` so it inherits
   the primary series' x-coordinate mapping.

`useMetricTrendState` is extended:

- Takes `annotations: Map<string, Annotation[]>` and `categories: Map<string, NoteCategory>` as inputs.
- Exposes a new `notesVisible` boolean and `toggleNotesVisible` callback.
- Passes the markPoint to `buildChartOption` only when `notesVisible`.

`MetricTrendBlock`:

- Fetches annotations for the asset+slo via a new `useTrendAnnotations(asset, slo)`
  hook (composed query — one request per trend view, not per point).
- Header gains a `MessageSquareWarning` toggle button next to `TargetDropdown`.
  Active state styled like the existing target toggle.
- On markPoint click, scrolls to the notes section and highlights the matching
  run's notes for ~1 s (CSS transition on `.highlight-note`).

### Override ring (separate from category system)

- `TrendPoint` domain gains `overridden: boolean` (mapper reads backend's
  `original_outcome` non-null signal).
- `buildChartOption`'s main-series symbol styling:
  - Currently renders selected points with a white border.
  - Extend: if `overridden && !isSelected(p)`, border is `ct.muted` (gray),
    width 2, symbolSize +2.
  - Selected-and-overridden → white border wins (selection is more important).
- Tooltip formatter appends `Override: {originalOutcome} → {outcome}` when
  `overridden`.

Override visual is **always on**. The per-chart notes toggle only controls the
markPoint pins.

## Dev seeding

`scripts/seed_evaluations.py` must be updated (or a new helper added that is
called from `dev-start.sh`):

- After evaluations are seeded, insert ~2–3 annotations across a handful of
  runs, covering every non-system category: `info`, `failure`, `investigation`.
- Insert one `re-evaluation` and one `override` annotation on specific runs to
  verify the system categories are filtered out of charts.

This ensures the trend charts show pins in dev without manual data creation.

  ECharts option passed to `ReactECharts`.
- `AddNoteForm`: dropdown lists categories alphabetically, defaults to `info`,
  submits `category_id` not free text.
- `CategoryForm`: validates label length, palette enum, unique name.
- Override ring: overridden trend point renders with gray border; selected
  overridden point renders with white border.

No integration tests required beyond the standard repository tests for the
new entity and FK.

## Rollout

No prod users → no staged rollout. Single branch, single PR sequence:

1. Backend: models, repositories, schemas, routers, seed migration, tests.
2. Seed-script update.
3. UI: `features/note-categories` + `CategoryManagementPage`.
4. UI: `AddNoteForm` + `NoteEntry` category refactor.
5. UI: `MetricTrendBlock` markPoint + override ring.

Step 1 must land first; subsequent steps can land independently.
