# SLO Format Redesign & Python Client Refactor

**Date:** 2026-03-15
**Status:** Approved

## Summary

Two related improvements driven by the same root cause â€” the original SLO format was ported
directly from Keptn 1.0 and carries legacy baggage (raw YAML string embedding, `spec_version`,
indicators inside SLOs). This spec covers:

1. **SLO format redesign** â€” full-stack change from opaque `slo_yaml: str` to structured
   `objectives` + `total_score` + `comparison` at every layer (DB, engine, API, client, manifests)
2. **Python client refactor** â€” pythonic fixes (imports at top, pydantic models, cleaner error
   handling) and a new `export` CLI command

## Architecture

PostgreSQL is the canonical source of truth. Manifest YAML is an import/export format only â€”
no files are stored on disk by the service. Users can query and (if needed) directly edit the DB.

## Section 1: Database Schema

### `slo_definitions` (modified)

Drop `slo_yaml TEXT`. Add flat columns for score and comparison.

```
id                      UUID PK
name                    TEXT NOT NULL
version                 INT  NOT NULL          -- auto-incremented per name
active                  BOOL NOT NULL DEFAULT true
display_name            TEXT
notes                   TEXT
author                  TEXT
meta                    JSONB NOT NULL DEFAULT '{}'
total_score_pass_pct    FLOAT NOT NULL DEFAULT 90.0
total_score_warning_pct FLOAT NOT NULL DEFAULT 75.0
comparison              JSONB NOT NULL DEFAULT '{}'  -- deserializes to SLOComparison via model_validate
created_at              TIMESTAMPTZ NOT NULL DEFAULT now()
```

Unique constraint: `(name, version)`. Indexes: `name`, `(name, version DESC)`.

`comparison` deserialization: `SLOComparison.model_validate(row.comparison)`. An empty dict
`{}` produces all-default values (`compare_with=SINGLE_RESULT`, `number_of_comparison_results=3`,
etc.) because every field on `SLOComparison` has a default.

### ORM classes (`api/app/db/models.py`)

Add `SLOObjective` ORM class in `models.py` alongside `SLODefinition`:

```python
class SLOObjective(Base):
    __tablename__ = "slo_objectives"

    id:                UUID PK
    slo_definition_id: UUID FK â†’ SLODefinition.id  (ondelete="CASCADE")
    sli:               Text NOT NULL
    display_name:      Text NOT NULL  server_default=""
    weight:            Integer NOT NULL  server_default=1
    key_sli:           Boolean NOT NULL  server_default=false
    sort_order:        Integer NOT NULL
    pass_criteria:     ARRAY(Text) NOT NULL  server_default="{}"
    warning_criteria:  ARRAY(Text) NOT NULL  server_default="{}"
```

Add relationship on `SLODefinition`:

```python
objectives: Mapped[list[SLOObjective]] = relationship(
    "SLOObjective",
    order_by="SLOObjective.sort_order",
    cascade="all, delete-orphan",
    lazy="selectin",
)
```

### `slo_objectives` (new table)

One row per objective per SLO version. Linked by `slo_definition_id` FK.

```
id                UUID PK
slo_definition_id UUID FK â†’ slo_definitions.id  ON DELETE CASCADE
sli               TEXT NOT NULL        -- references SLI definition name at evaluation time
display_name      TEXT NOT NULL DEFAULT ''
weight            INT  NOT NULL DEFAULT 1
key_sli           BOOL NOT NULL DEFAULT false
sort_order        INT  NOT NULL        -- preserves user-defined order; 0-based
pass_criteria     TEXT[] NOT NULL DEFAULT '{}'   -- e.g. {"<500","<=+10%"}
warning_criteria  TEXT[] NOT NULL DEFAULT '{}'
```

Criteria are flat `TEXT[]`. The Keptn-inherited multi-block AND/OR structure (list of
`SLOCriteria` each containing `list[str]`) is dropped â€” it was never used in practice.

SLO definitions are **immutable** â€” new objectives are always inserted alongside a new
`slo_definitions` version. Objectives are never updated in place.

### `evaluations` (modified)

Drop `slo_yaml TEXT` column. It is nullable and redundant â€” `slo_name + slo_version` already
reference the immutable `slo_definitions` row, from which the full SLO can always be
reconstructed. The evaluation worker loads the SLO from DB at job execution time using
`slo_name + slo_version`, rather than reading a snapshotted YAML string.

`EvaluationRepository.mark_completed()`: remove the `slo_yaml: str` parameter and the
corresponding `.values(slo_yaml=slo_yaml, ...)` assignment. No replacement â€” the column
is gone. Update the caller (evaluation worker) to remove the `slo_yaml=...` argument.

### Migration

Autogenerated Alembic migration (single migration file):
- Drop `slo_definitions.slo_yaml`
- Add `slo_definitions.total_score_pass_pct`, `total_score_warning_pct`, `comparison`
- Create `slo_objectives` table
- Drop `evaluations.slo_yaml`

**Existing data:** This is a clean break. If a developer has existing rows in `slo_definitions`
or `evaluations.slo_yaml`, they must truncate those tables before running the migration, or
manually run `UPDATE slo_definitions SET ... WHERE ...` to populate the new columns. Document
this in the migration file as a comment. No production deployment exists yet.

## Section 2: Engine

### `SLO` model (`slo_models.py`)

Drop `spec_version` (redundant with `api_version` in manifests) and `indicators` (indicators
live in the SLI definition, not the SLO).

Criteria flatten from `list[SLOCriteria]` to `list[str]`:

```python
class SLOObjective(BaseModel):
    sli: str
    display_name: str = ""
    pass_criteria: list[str] = []
    warning_criteria: list[str] = []
    weight: int = 1
    key_sli: bool = False

class SLO(BaseModel):
    objectives: list[SLOObjective]
    comparison: SLOComparison
    total_score: SLOTotalScore
```

`SLOCriteria` class is deleted â€” no longer needed.

`SLOParseError` is kept and repurposed: rename module docstring, but keep the class for
validation errors raised by `build_slo` (e.g. empty objectives list, unknown keys in input
dict). `build_slo` raises `SLOParseError` for invalid input rather than letting pydantic
`ValidationError` surface directly to callers.

### `slo_parser.py`

Repurposed from YAML text parser to a constructor. Rename `parse_slo` to `build_slo`:

```python
def build_slo(
    objectives: list[dict[str, Any]],
    total_score_pass_pct: float,
    total_score_warning_pct: float,
    comparison: dict[str, Any],
) -> SLO:
```

Constructs and returns a validated `SLO` model. `objectives` is a list of dicts matching
`SLOObjective` field names (as returned from DB rows or API payload). No YAML parsing.
Validation that objectives reference known indicators is removed from this layer â€” it moves
to `AssetSLOLink` creation where both SLI and SLO are in scope.

### `evaluator.py`

Two changes:

1. Signature: `evaluate(slo_yaml: str, ...)` â†’ `evaluate(slo: SLO, ...)`. The
   `parse_slo(slo_yaml)` call at the top of the function is removed; `slo` is used directly.

2. `_build_targets`: currently iterates `for block in objective.pass_criteria` then
   `for raw in block.criteria`. With flat `list[str]`, this becomes a single loop:
   `for raw in objective.pass_criteria`. Update both `pass_criteria` and `warning_criteria`
   paths in `_build_targets`.

No other logic changes inside the evaluator.

## Section 4: Client + Manifest

### `manifest.py`

All internal types converted from `@dataclass` to `pydantic.BaseModel`. Mutable default
fields use `Field(default_factory=list)`:

```python
class ApplyPlan(BaseModel):
    actions: list[PlanAction] = Field(default_factory=list)

class ApplyResult(BaseModel):
    created: int = 0
    updated: int = 0
    skipped: int = 0
    failed: int = 0
    errors: list[ApplyError] = Field(default_factory=list)
```

`ManifestDocument`, `PlanAction`, `ApplyError` have no mutable defaults â€” plain `BaseModel`
fields suffice.

`_has_diff()` for `SLO` kind: order-sensitive structural equality. Compare all four fields:
- `doc.spec.get("objectives")` vs `[o.model_dump(exclude={"sort_order"}) for o in existing.objectives]`
- `doc.spec.get("total_score", {}).get("pass_pct")` vs `existing.total_score_pass_pct`
- `doc.spec.get("total_score", {}).get("warning_pct")` vs `existing.total_score_warning_pct`
- `doc.spec.get("comparison", {})` vs `existing.comparison`

A changed `comparison` triggers a new version, same as changed objectives or score.

`_diff_reason()` for `SLO`: return `"objectives or score differ (new version will be created)"`.

`_create()` for `SLO` kind: extract and pass structured fields from `doc.spec`:
```python
client.slo_definitions.create(
    name,
    objectives=doc.spec["objectives"],
    total_score_pass_pct=doc.spec.get("total_score", {}).get("pass_pct", 90.0),
    total_score_warning_pct=doc.spec.get("total_score", {}).get("warning_pct", 75.0),
    comparison=doc.spec.get("comparison", {}),
    display_name=doc.metadata.get("display_name"),
    notes=doc.metadata.get("notes"),
    author=doc.metadata.get("author"),
)
```

`_update()` for `SLO` kind: identical call â€” SLO updates create a new version, same as
`_create()`. No delete+recreate needed (unlike `AssetSLOLink`).

### `models.py`

`SLODefinition` drops `slo_yaml`, gains:
- `objectives: list[dict[str, Any]]`
- `total_score_pass_pct: float`
- `total_score_warning_pct: float`
- `comparison: dict[str, Any]`

### `cli.py`

- Inline imports inside `apply()` function body moved to top of file
- `contextlib.suppress(Exception)` in `_raise_for_status` replaced with explicit try/except:

```python
def _raise_for_status(resp: httpx.Response) -> None:
    if resp.is_success:
        return
    try:
        data = resp.json()
        detail = data.get("detail", resp.text) if isinstance(data, dict) else resp.text
    except Exception:
        detail = resp.text
    match resp.status_code:
        case 404:
            raise TropekNotFoundError(detail)
        case 409:
            raise TropekConflictError(detail)
        case 422:
            raise TropekValidationError(detail)
        case _:
            raise TropekAPIError(resp.status_code, detail)
```

- New `export` command added (see below)

### New manifest format for `SLO` kind

```yaml
api_version: tropek/v1
kind: SLO
metadata:
  name: http-availability-slo
  display_name: HTTP Availability SLO
  author: bootstrap
  notes: Covers P99 latency, error rate, and availability for HTTP services.
spec:
  total_score:
    pass_pct: 90.0
    warning_pct: 75.0
  objectives:
    - sli: response_time_p99
      display_name: "Response Time P99"
      pass_criteria: ["<500"]
      warning_criteria: ["<800"]
      weight: 2
    - sli: error_rate
      key_sli: true
      pass_criteria: ["<0.01"]
      warning_criteria: ["<0.05"]
      weight: 3
```

No `slo_yaml`, no `spec_version`, no `indicators`.

### New `export` CLI command

```
tropek export [-f output.yaml] [--base-url URL] [--api-key KEY]
```

Calls these existing GET endpoints in `_KIND_ORDER` order:
1. `GET /asset-types` â†’ `AssetType` documents
2. `GET /datasources` â†’ `DataSource` documents
3. `GET /assets` â†’ `Asset` documents
4. `GET /sli-definitions` â†’ `SLI` documents
5. `GET /slo-definitions` â†’ `SLO` documents
6. `GET /asset-groups` â†’ `AssetGroup` documents
7. For each asset: `GET /assets/{name}/slo-links` â†’ `AssetSLOLink` documents
8. For each group: `GET /asset-groups/{name}/slo-links` â†’ `AssetGroupSLOLink` documents

Writes multi-document YAML (separated by `---`) to `-f` path or stdout. No new API endpoints
required.

**SLO kind reconstruction** (non-obvious mapping):
- `total_score_pass_pct` / `total_score_warning_pct` â†’ nested under `spec.total_score.pass_pct`
  / `spec.total_score.warning_pct`
- `objectives` list (excluding `sort_order`) â†’ `spec.objectives` in `sort_order` order
- `display_name`, `notes`, `author` â†’ `metadata` block
- `comparison` â†’ `spec.comparison` (omit if `{}`)

All other kinds have 1:1 field mapping between API response and manifest spec.
